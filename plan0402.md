# 2026-04-02 下一步优化路线（AO / 大气湍流遥感恢复）

## 1) 模型方向：从“单模型”走向“分阶段 + 可解释约束”

1. **主干建议：Diffusion / Rectified Flow + 物理先验**
   - 当前任务中，退化过程具有明确物理结构（波前畸变、PSF、噪声、压缩伪影）。
   - 建议在生成模型中显式加入“前向退化一致性约束”（degradation consistency），并对不同退化强度分桶训练。
   - 对推理时延敏感场景，可优先测试 **Rectified Flow / consistency distillation** 以减少采样步数。

2. **高性价比路线：UNet/NAFNet/Restormer 系列判别式模型**
   - 若目标是工程可落地与推理速度，建议优先试验：
     - NAFNet（高效、稳定）
     - Restormer / Uformer（长程依赖更强）
     - SwinIR（在遥感纹理恢复上常见）
   - 作为第一阶段恢复器，可在后端叠加迭代物理优化模块。

3. **新增 VAE 的定位**
   - VAE 更适合作为：
     - 轻量可部署 baseline；
     - latent 空间不确定性建模模块；
     - 与 diffusion 的 teacher/student 组合（VAE 负责结构先验，diffusion 精修细节）。

4. **GAN 的后续定位**
   - GAN 可保留为“纹理增强头”而非唯一主模型：
     - stage1: 保真恢复（L1/Charbonnier + 物理约束）
     - stage2: 对抗细节增强（小权重、短轮次）
   - 优先避免 GAN 全流程主导导致的不稳定。

## 2) 训练策略优化（建议按优先级执行）

1. **两阶段/三阶段训练**
   - 阶段A：仅重建损失（稳定收敛）
   - 阶段B：加入物理一致性损失
   - 阶段C：可选加入对抗损失（低学习率、短周期）

2. **课程学习（curriculum）**
   - 先训练轻退化样本，再逐步提高湍流强度、模糊核大小、噪声水平。
   - 能显著降低早期训练不稳定与模式崩塌风险。

3. **损失函数建议**
   - 重建项：L1/Charbonnier + MS-SSIM（可选）
   - 感知项：LPIPS / VGG perceptual（低权重）
   - 物理项：前向退化一致性（当前已具备基础）
   - 对抗项：Hinge GAN 或 least-squares GAN（通常比 BCE 更稳）

4. **优化器与调度**
   - 试验 AdamW + cosine decay + warmup。
   - GAN 使用 TTUR（D 学习率略高或更新步数更高），并配合 label smoothing / R1 regularization。

## 3) 是否需要 downstream 迭代/计算优化：建议“需要”

1. **Plug-and-Play / Unrolled Optimization（推荐）**
   - 将网络输出作为先验，结合物理退化算子进行 3~10 步迭代修正：
     - x_{k+1} = x_k - \alpha \nabla(\|H(x_k)-y\| + \lambda R_{net}(x_k))
   - 优点：可解释、对分布外退化更鲁棒。

2. **测试时自适应（Test-Time Adaptation）**
   - 对单场景短步微调（仅 BN/小头参数）或进行 self-ensemble。
   - 对遥感真实域偏移（传感器/天气变化）效果明显。

3. **后处理与约束**
   - 加入频域一致性检查（MTF 约束）或边缘保持正则。
   - 对 AO 场景可考虑波前参数反演支路，形成“恢复 + 估计”多任务。

## 4) 数据与评估体系升级

1. **退化参数标注与分桶评估**
   - 在数据管线记录退化元数据（湍流强度、噪声、运动模糊）并按桶报告指标。
   - 避免单一平均 PSNR 掩盖模型短板。

2. **增加真实域验证集**
   - 至少准备 small real set（即便无 GT，也可做人类主观评分/无参考指标）。

3. **评估指标建议**
   - 保留 PSNR/SSIM/LPIPS；
   - 新增下游任务指标（例如遥感检测/分类 mAP 或 OA）验证“恢复是否真正有用”。

## 5) 工程与性能优化

1. **GPU 利用率优化**
   - 训练阶段减少高频 CPU 指标计算（已在 fast_train 体现）；
   - 指标计算尽量迁移到 torch 实现，减少 `.cpu().numpy()` 往返。

2. **吞吐与稳定性**
   - 启用 AMP、pin_memory、persistent_workers、prefetch_factor（当前已具备）；
   - 可进一步尝试 gradient accumulation 与 channels-last。

3. **实验管理**
   - 推荐引入统一实验追踪（W&B / TensorBoard）记录损失分量、梯度范数、退化分桶指标。

## 6) 建议的 2~3 周执行顺序

1. Week 1：
   - 跑通 UNet/Restormer/NAFNet 三个判别式 baseline；
   - 建立分桶评估与统一日志。

2. Week 2：
   - 引入两阶段训练（重建 -> 物理一致性）；
   - 在最佳判别式模型上接 3~5 步 downstream 迭代修正。

3. Week 3：
   - 对比 GAN 精修与 diffusion/refiner 精修；
   - 输出最终“质量-时延-稳定性”三维权衡报告并选型。

---

## 一句话结论

- 若目标是**稳定可部署**：优先 NAFNet/Restormer + 物理一致性 + 少步迭代修正。  
- 若目标是**极致感知质量**：判别式主干 + diffusion/refiner（或轻量 GAN）两阶段方案更稳妥。




这三种落盘（存储）结构在计算机视觉（尤其是视频/多帧序列任务）中非常经典，它们分别对应了**数据处理的不同阶段和不同规模**。

针对你的大气湍流修复任务（需要输入多帧模糊序列，输出单张清晰遥感图），我为你详细对比它们的优劣，并在最后给出具体的选择建议。

---

### 1. A: 图像序列文件夹 (每个样本一个目录)
*   **格式特征**：最原始、最直观的文件系统结构。例如 `sample_0001/` 目录下存放 `clean.png` 和 `frame_000.png` 到 `frame_006.png`。
*   **优势 (Pros)**：
    *   **所见即所得，极易调试**：你可以直接在文件管理器中双击点开图片，肉眼检查物理仿真器生成的湍流效果对不对、有没有黑边、有没有生成失败。
    *   **修改极度灵活**：如果某几张图片生成坏了，直接删掉或者替换，不需要重写整个数据库。
    *   **代码门槛低**：用最基础的 `os.listdir` 配合 `cv2.imread` 或 `PIL.Image.open` 就能写出 Dataloader。
*   **劣势 (Cons)**：
    *   **致命的 I/O 瓶颈（小文件灾难）**：假设你有 5万个 Patch，每个 Patch 有 1+7=8 张图，总共就是 40万张小图片。在训练阶段，每个 Epoch 你的硬盘都要频繁地“打开文件->读取->关闭文件”几十万次。无论你的显卡有多快，都会因为 CPU 和硬盘 I/O 读不过来而导致 GPU 处于“饿死”状态（利用率极低）。
    *   **容易打爆 Linux inode**：如果未来数据量达到百万级，会把服务器系统的文件索引节点（inode）耗尽。

### 2. B: LMDB / WebDataset (打包成键值或tar分片)
*   **格式特征**：将成千上万个小文件打包成少数几个大文件。读取时通过内存映射（LMDB）或流式顺序读取（WebDataset）。目前超大规模视觉模型训练的**业界标配**。
*   **优势 (Pros)**：
    *   **起飞的 I/O 性能**：彻底解决了小文件读取的寻道延迟问题。LMDB 可以将整个大文件映射到内存中（Memory-mapped），极大地提升了 Dataloader 的吞吐量，让 GPU 满载运行。
    *   **易于在集群上共享/移动**：拷贝 1 个 50GB 的 LMDB 文件，比拷贝 40万个几十KB的图片要快几百倍。
*   **劣势 (Cons)**：
    *   **黑盒，无法直接预览**：打包后，你没法直接双击看图片了，必须写专门的 Python 脚本从数据库里读取并保存出来才能看。
    *   **修改困难**：如果想在中间插入或删除几个样本，通常需要把整个数据库重新打包一遍。
    *   **上手有一定门槛**：需要学习相关的库（如 `lmdb` 或 `webdataset`）的打包和读取代码。

### 3. C: NumPy/PT 张量 (保存为 .npz / .pt)
*   **格式特征**：跳过图像编码（JPG/PNG），直接把经过预处理的像素矩阵保存为 PyTorch 张量或 NumPy 数组。
*   **优势 (Pros)**：
    *   **省去了图像解码的 CPU 算力**：训练时直接载入内存就是 Tensor，不需要经历 `Image -> Decode -> ToTensor` 的过程。
    *   **可以保存超越 0-255 范围的物理量**：如果你的物理仿真器输出了高精度的浮点数（比如某些真实的大气相位屏矩阵，或者包含负数的波前数据），传统 PNG 存不下，用 `.pt` 或 `.npy` 最合适。
*   **劣势 (Cons)**：
    *   **极度浪费硬盘空间**：未压缩的 Tensor（特别是 float32）体积非常庞大，可能比 PNG 大十几倍。你的硬盘很快就会被塞满。
    *   **依然存在小文件 I/O 灾难**：如果每个样本存一个 `.pt`，相当于 Option A 的体积膨胀版，读取瓶颈依然存在。

---

### 💡 最终选择建议：采用“两步走”策略 (A -> B)

对于你当前处于“起步并验证”阶段的科研项目，**绝对不要一开始就选 B 或 C，而应该：**

#### **第一阶段：数据集生成与调试期 👉 毫无疑问选 A (图像序列文件夹)**
1. 你现在还在测试物理仿真代码，**可视化验证（Visual Check）是第一要务**。你需要直观地看到 2100 张遥感图片被裁切、模拟出湍流后的样子。
2. 先用 **A结构** 生成一个小规模的数据集（比如只生成 2000 个 Patch 序列）。
3. 用这个小数据集写 Dataloader，把你的模型（如 UNet / SwinIR）跑通，验证 loss 能不能下降。这时候因为数据量小，小文件的 I/O 瓶颈还显现不出来。

#### **第二阶段：全量训练与冲刺期 👉 将 A 转换为 B (LMDB)**
1. 当你验证了模型有效，并且决定用 DOTA 或 NWPU 等大数据集生成几万、几十万个序列时，A 结构会让你的训练速度慢如蜗牛。
2. 此时，写一个 Python 转换脚本，**把 A 结构中的所有图片打包成 B (LMDB 格式)**。
3. Github 上开源的底层视觉任务库（如去噪、超分的经典框架 **BasicSR**），几乎都内置了把文件夹（Option A）打包成 LMDB（Option B）的脚本（`create_lmdb.py`），你可以直接拿来用。
4. 训练时，让 Dataloader 直接从 LMDB 里读取数据，你的训练速度将有质的飞跃。

**关于选项 C**：除非你的物理仿真输出的不仅是人眼可见的 RGB 图像，还包含必须以 float32 保存的物理中间变量（如光程差矩阵 OPD），否则**通常不推荐**大批量使用 C。
